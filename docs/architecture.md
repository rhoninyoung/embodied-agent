# 系统架构设计

## 总体设计哲学

本系统采用 **分层解耦** 架构。AI Agent（运行在云端/Cursor中）负责高层决策，本地中间件负责实时控制，两者通过明确定义的接口通信。这样既利用了 AI 的强大推理能力，又满足了物理操作对实时性的要求。

```
┌──────────────────────────────────────────────────────────────┐
│                    Layer 4: AI Agent (决策层)                  │
│                                                              │
│  Cursor / Claude  ──  理解意图、设计零件、生成控制指令          │
│  交互模式: 请求-响应 (延迟 ~1-5s)                              │
└──────────────────────┬───────────────────────────────────────┘
                       │  WebSocket / REST API
┌──────────────────────▼───────────────────────────────────────┐
│                Layer 3: 本地中间件 (桥接层)                     │
│                                                              │
│  Python 服务  ──  实时控制循环、安全边界、状态管理               │
│  职责:                                                        │
│    - 将 AI 高层指令翻译为底层控制信号                           │
│    - 维护设备状态机                                            │
│    - 实施安全约束（力限制、工作空间边界、急停）                   │
│    - 图像采集与预处理                                          │
```

## 各层详细设计

### Layer 4: AI Agent — 决策核心

**角色**: 系统的"大脑"，负责所有需要理解、推理、创造的任务。

**能力范围**:

| 能力 | 说明 |
|------|------|
| 任务理解 | 解析自然语言指令，拆解为可执行步骤 |
| 3D 建模 | 生成 OpenSCAD 脚本 → 编译为 STL |
| 固件编写 | 为 Arduino/ESP32 生成控制固件 |
| G-code 生成 | 通过切片参数控制打印质量 |
| 视觉分析 | 分析摄像头图像，识别零件、检测装配质量 |
| 运动规划 | 生成机械臂的路径点序列 |
| 异常处理 | 根据传感器反馈判断并调整策略 |

**通信协议**: 通过 WebSocket 与本地中间件双向通信

```json
// AI → 中间件: 机械臂控制指令
{
  "type": "command",
  "target": "arm",
  "action": "move_to",
  "params": {
    "x": 150.0, "y": 0.0, "z": 100.0,
    "speed": 50,
    "grip": "open"
  }
}

// AI → 中间件: 按需拍照 (L0 模式)
{
  "type": "command",
  "target": "vision",
  "action": "capture",
  "params": {
    "resolution": "1080p",
    "ptz": {"pan": -30, "tilt": 0}
  }
}

// AI → 中间件: 切换视觉交互级别
{
  "type": "command",
  "target": "vision",
  "action": "set_vision_level",
  "params": {
    "level": "L1",
    "interval_sec": 10
  }
}

// 中间件 → AI: 图像上报
{
  "type": "vision_frame",
  "source": "camera",
  "trigger": "on_demand",
  "image_base64": "<base64 encoded jpeg>",
  "resolution": [1920, 1080],
  "timestamp": 1709568000
}

// 中间件 → AI: 状态上报
{
  "type": "status",
  "source": "arm",
  "state": "idle",
  "position": {"x": 150.0, "y": 0.0, "z": 100.0},
  "grip_force": 0.5,
  "timestamp": 1709568000
}
```

### Layer 3: 本地中间件 — 实时桥接

**技术栈**: Python 3.10+ / asyncio

**核心模块**:

```
middleware/
├── server.py          # WebSocket 服务端，对接 AI Agent
├── device_manager.py  # 设备生命周期管理
├── safety.py          # 安全约束引擎
├── state_machine.py   # 系统状态机
└── config.yaml        # 设备配置
```

**安全约束引擎** (最关键的模块):

- **工作空间限制**: 机械臂运动范围的虚拟围栏
- **力矩限制**: 超过阈值立即松开
- **速度限制**: 靠近物体时自动减速
- **急停机制**: 硬件按钮 + 软件触发双重保护
- **指令校验**: 所有 AI 指令在执行前经过合法性检查

### Layer 2: 设备驱动层

#### 视觉模块

```
vision/
├── camera.py          # 摄像头驱动 (OpenCV)
├── ptz_control.py     # 云台控制 (串口/USB)
├── capture.py         # 图像采集与预处理
└── calibration.py     # 相机标定
```

- 图像分辨率: 1080p (精细操作时可切换 4K)
- **本地采集帧率**: 5fps（供中间件本地处理，如运动检测、变化检测），精细操作时 30fps
- **上报 AI 频率**: 取决于视觉交互级别（见下方策略），Phase 0 默认按需快照
- 预处理: 畸变校正、白平衡、裁剪感兴趣区域

#### 视觉交互策略（Token 经济性设计）

AI Agent 每接收一张 1080p 图像约消耗 1,000\~2,000 tokens。为控制成本，系统采用分级策略，中间件在本地完成高频采集与低级视觉处理，仅在"有意义"时才将图像上报给 AI：

| 级别 | 模式 | 触发方式 | 上报频率 | Token 消耗 | 适用场景 |
|------|------|---------|---------|-----------|---------|
| **L0** | 按需快照 | AI 主动发起 `capture` 指令 | 仅在请求时 | 最低 | 环境概览、初期探索 |
| **L1** | 事件驱动 | 中间件检测到显著变化后上报 | ~0.1fps | 低 | 操作结果确认、异常告警 |
| **L2** | 低速轮询 | 定时上报（可配置间隔） | 0.1\~0.2fps | 中等 | 打印监控、长时间任务 |
| **L3** | 关键帧上报 | 本地 30fps 采集，智能选取关键帧 | ~1\~5fps | 较高 | 精密装配、焊接引导 |

**级别切换规则**:

- 系统启动默认 **L0**（按需快照）
- AI 可通过指令动态切换级别：`{"action": "set_vision_level", "level": "L1"}`
- 中间件可在检测到异常时临时升级（如检测到碰撞 → 自动切到 L1 上报现场图像）
- 每次级别变更写入日志，便于事后分析 token 消耗

**L1 事件触发条件**（中间件本地 OpenCV 判断）:

- 帧间差异超过阈值（场景发生显著变化）
- 机械臂动作执行完成（操作结果确认）
- 传感器异常（力过大、位置偏差超限）
- 打印状态变更（开始/暂停/完成/失败）

**Token 消耗估算参考**:

| 级别 | 每分钟图片数 | 估算 Token/分钟 | 每小时估算 |
|------|------------|----------------|-----------|
| L0 | 1\~5（按需） | 1K\~10K | 60K\~600K |
| L1 | ~6 | ~6K\~12K | ~360K\~720K |
| L2 | 6\~12 | ~6K\~24K | ~360K\~1.4M |
| L3 | 60\~300 | ~60K\~600K | ~3.6M\~36M |

#### 机械臂控制模块

```
control/
├── arm_driver.py      # 串口通信驱动
├── kinematics.py      # 正/逆运动学
├── trajectory.py      # 轨迹规划与插值
├── gripper.py         # 夹爪控制
└── servo_config.yaml  # 舵机参数配置
```

- 通信协议: UART (115200 baud) 或自定义串口协议
- 控制方式: 关节空间插值 + 笛卡尔空间运动
- 反馈: 舵机电流/角度反馈 → 估算末端力

#### 3D 打印控制模块

```
printer/
├── bambu_client.py    # 拓竹局域网 MQTT 客户端
├── slicer.py          # 调用切片引擎 (PrusaSlicer CLI)
├── model_gen.py       # OpenSCAD → STL 管线
└── print_monitor.py   # 打印状态监控
```

- 拓竹 P2S 支持局域网 MQTT 协议，可直接发送打印任务
- 切片: 通过 PrusaSlicer CLI 从 STL 生成 G-code
- 建模: AI 生成 OpenSCAD 脚本 → 编译输出 STL

## 数据流示例: "打印并安装一个夹爪"

```
1. 用户: "给机械臂做一个新的夹爪"
2. AI: 分析当前夹爪接口尺寸 (请求摄像头拍照)
3. 中间件: 控制云台对准机械臂末端，拍照回传
4. AI: 识别接口规格，用 OpenSCAD 设计新夹爪
5. AI: 生成 STL → 调用切片 → 发送打印任务
6. 中间件: 监控打印进度，完成后通知 AI
7. AI: 规划拆卸旧夹爪、安装新夹爪的动作序列
8. 中间件: 逐步执行动作，每步拍照确认
9. AI: 验证安装结果，测试夹爪开合
```

## 技术选型一览

| 组件 | 技术方案 | 备选 |
|------|---------|------|
| AI Agent | Claude (via Cursor) | 本地 LLM (llama.cpp) |
| 中间件 | Python asyncio + FastAPI | Node.js |
| 视觉 | OpenCV + USB Camera | RealSense 深度相机 |
| 3D 建模 | OpenSCAD (脚本化) | CadQuery (Python) |
| 切片 | PrusaSlicer CLI | Bambu Studio CLI |
| 打印通信 | MQTT (局域网) | FTP + 拓竹 API |
| 机械臂通信 | pyserial (串口) | ROS2 |
| 前端监控 | Web Dashboard (可选) | 纯终端 |
